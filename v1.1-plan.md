# Active Graph v1.1 — consolidated backlog

Generated by the post-v1.0.3 contract review (2026-05-19). This
plan supersedes the scattered "filed for v1.1" markers in
CONTRACT.md amendments from v0.5 onward. CONTRACT.md now points
here instead of carrying inline forward-references.

This is a **backlog, not a roadmap**. Priority tiers help
sequencing; "v1.1 must include X" is not the framing. The plan
collects every concrete v1.1 candidate raised across milestone
amendments and assigns each: title, description, source amendment,
dependencies, priority tier, scope estimate.

CONTRACT v1.1 expands this into a proper milestone scope when v1.1
work begins. Until then, the plan is the visibility layer for what
is owed.

---

## Open questions before v1.1 work begins

These bottleneck the consolidation pass — answer before scoping
the milestone:

1. **Is `graph.query` deprecated in v1.1, or carried indefinitely?**
   v1.0.3 #1 carved out "no deprecation in v1.0.3." The choice
   shapes API-DEP-1 below.
2. **Is `Runtime.load` auto-provider a "v1.1 must" or a "v1.x maybe"?**
   The rc1 user-test surfaced friction (B2); the rc2 fix was
   tactical. The decision shapes RUNTIME-1 below.
3. **CONTRACT modularization (DOC-1 / §7 #5 of findings):**
   split-per-milestone, executive-summary preambles, or no
   change? The decision is process; it does not block code work
   but shapes the v1.1 amendment-cycle workflow.
4. **Test-file naming sweep (§7 #6 of findings):** rename
   `test_v1_0_3_*` and similar version-tagged files? Cosmetic;
   landing in v1.1 is the natural moment.

## Priority tiers

Each entry carries one of:

- **must-have** — closes a documented adoption-surface gap; the
  framework's coherence depends on it.
- **should-have** — closes a discipline gap or a deferred
  follow-on with shipped fixes in adjacent work.
- **nice-to-have** — ergonomics, harmonization, or polish whose
  absence does not block adoption.

Tiers are review-time assessments, not commitments. v1.1 scoping
re-prioritizes against capacity.

## Backlog by theme

### Theme A — Error-completeness milestone (the largest, oldest backlog)

The v1.0 PR-A-through-PR-G series ended with ~30 items deferred.
Final tally per PR-G end-of-series note (CONTRACT v1.0 §PR-G):
22 partial Pack* migrations, 4 DB-error wrappers, plus the rc1
user-test gate carryover (now superseded by shipped rc2/v1.0.1
work). Re-applying the events-not-exceptions principle (CONTRACT
v1.0 #4b) is part of the milestone — some partials may turn out to
be event-shaped on second look.

#### A-1 — Complete the 22 partial Pack* migrations

**Description**: 22 of 28 Pack* registration-error raise sites are
unmigrated — they use the legacy single-message
`ActiveGraphError.__init__` branch (format-noncompliant but valid).
PR-E migrated only the 2 high-value `loader.py` sites. The classes
are in the v1.0 hierarchy; messages catch up incrementally.

**Source**: CONTRACT v1.0 §PR-E "Partial migration note"; restated
in §PR-F, §PR-G "v1.1 error-completeness milestone scope."

**Dependencies**: none.

**Priority**: should-have. Each unmigrated raise produces a
format-noncompliant error message; users hitting one of the 22
sites get the v0.9-shaped message instead of the v1.0 structured
form.

**Scope**: medium. Mostly mechanical, but each migration involves a
"what's the actual recovery prose" decision per site. Reverse-audit
order (hardest first) per the v1.0 PR-E discipline.

#### A-2 — Migrate the 4 deferred DB-error wrappers

**Description**: SQLite `sqlite3.OperationalError` and Postgres
`psycopg.OperationalError` variants. Per CONTRACT v1.0 §PR-C
flagged-not-migrated: multiple distinct failure modes (lock
timeout, journal corruption, disk full, auth, host unreachable,
db missing, conn dropped). Each needs per-mode recovery prose.

**Source**: CONTRACT v1.0 §PR-C "Flagged-not-migrated"; restated
in §PR-G "v1.1 error-completeness milestone scope."

**Dependencies**: needs a contributor with hands-on DB-failure
experience; the PR-C discipline note rejected making up prose.

**Priority**: should-have. Without it, real production failures
bubble unwrapped driver exceptions.

**Scope**: medium-to-large. 6-10 distinct failure modes across
two drivers, each with its own recovery prose.

#### A-3 — Re-audit Pack* migrations under the events-not-exceptions principle

**Description**: per CONTRACT v1.0 §PR-G series-completion note,
some of A-1's 22 partial migrations may be event-shaped on second
look (e.g., `PackPromptLoadError` firing during pack reload after
the runtime is up may belong as `behavior.failed` rather than as
a raised exception). Apply CONTRACT v1.0 #4b as the audit instrument.

**Source**: CONTRACT v1.0 §PR-G series-completion note.

**Dependencies**: A-1 (re-audit happens before migration locks
in).

**Priority**: should-have.

**Scope**: small (audit), then folds back into A-1's scope per
finding.

### Theme B — Provider commitment surface (v1.0.1 / v1.0.2 / v1.0.3 carryovers)

#### B-1 — OpenAI tool-shape translation in `Tool.to_definition()`

**Description**: `Tool.to_definition()` emits the Anthropic
`{name, description, input_schema}` shape. OpenAI's chat-completions
API expects `{type:"function", function:{name, description,
parameters}}` and returns `tool_calls` rather than `tool_use`
content blocks. `OpenAIProvider.complete()` currently raises
`LLMBehaviorError(reason="llm.network_error")` with a v1.1 pointer
when `tools=` is non-empty.

v1.1 scope: provider-aware tool-shape translation (either
parameterize `Tool.to_definition(provider="anthropic"|"openai")`
or push the shape into the provider's `complete()`), parallel
`tool_calls` extraction in `OpenAIProvider`, parity tests across
both providers for the same tool-using behavior.

**Source**: CONTRACT v1.0.1 #5(c) clause 2; restated in v1.0.3 #4
"OpenAI tool support is out of scope" and in v1.1 #7-and-beyond.

**Dependencies**: none.

**Priority**: must-have. The current state ships two providers
with asymmetric tool support; that undermines v1.0.1 #5's
provider-commitment claim for tool-using workloads.

**Scope**: large. New `Tool.to_definition(provider=...)` shape,
new `OpenAIProvider` tool-extraction path, parity-test
infrastructure, error-semantics alignment with v0.6 #11 reason
taxonomy.

#### B-2 — Native structured-output mode for providers that support it

**Description**: both shipped providers use the framework's
instruction-based path (schema + example instance in system prompt
per v1.0.1 #2). OpenAI's `response_format={"type":"json_schema",...}`
mode forces the model to emit conforming JSON natively, eliminating
the schema-echo failure mode v1.0.1 #2 patched in the prompt.

v1.1 scope: opt-in flag on `@llm_behavior` (or per-provider config)
toggling native mode, cache-key impact analysis (prompt-hash inputs
change because the system prompt shrinks), replay-determinism
guarantees across the mode flip.

**Source**: CONTRACT v1.0.1 #5(c) clause 3; restated in v1.0.3
"v1.1 backlog candidates" and v1.1 #7-and-beyond.

**Dependencies**: B-1 desirable (same provider-aware codepath).

**Priority**: should-have. Closes a stated non-promise from
v1.0.1; not blocking adoption today.

**Scope**: medium. Mostly cache-key and replay-determinism
analysis; the API surface is small.

#### B-3 — Pack-load-time cross-provider validation

**Description**: pack behaviors loaded after Runtime construction
via `load_pack` currently fall through to first-run validation in
`_ensure_registry()` rather than load-pack-time validation. v1.0.2.post1
fired validation at the decorator-time and Runtime-construction
binding moments, but did not extend to `load_pack` because that
binding moment surfaces a different ordering.

**Source**: CONTRACT v1.0.2 #1(b); restated in v1.0.3 "v1.1
backlog candidates" #5.

**Dependencies**: none.

**Priority**: should-have. Closes the third binding moment named
in v1.0.2 #1(b). Today a pack with a cross-provider mismatch fails
lazily; users with packs see the lazy path.

**Scope**: small. The validator factored out in v1.0.2.post1's
`activegraph/runtime/_live.py` is reusable; the work is integrating
it into `load_pack`'s call path.

#### B-4 — Dict-form `output_schema=` support

**Description**: users reasonably pass JSON-schema dicts to
`output_schema=` because that's the de facto cross-provider
structured-output format. v1.0.3 #2 strict-validates at decoration
time (rejects dicts with a structured TypeError + example). v1.1
opens the design question of whether to accept dicts via a
runtime converter or via a dedicated `output_schema_dict=` kwarg.

Open design questions per CONTRACT v1.0.3 #2:
- Which providers' structured-output APIs accept dicts natively?
- What is the equivalence contract between dict-form and
  Pydantic-form schemas?
- What is the prompt-hash treatment for fixtures (the dict is
  hashable; the Pydantic class is not, today)?
- Should the framework grow `LLMProvider.recognizes_schema_dict()`
  as a Protocol method (mirroring v1.0.2 #1's
  `recognizes_model()`)?

**Source**: CONTRACT v1.0.3 #2 "v1.1 candidate" + "v1.1 backlog
candidates."

**Dependencies**: design pass needed before implementation.

**Priority**: should-have. v1.0.3 #2 closed the user's pain
(silent failure with confusing diagnostic) via early failure; the
ergonomics gain is real but not blocking.

**Scope**: large. The decision matrix above each maps to
implementation choices; the wrong choice ships a feature that
fights with the framework's existing schema model.

#### B-5 — `on_failure=` callback on `run_goal()` and siblings

**Description**: v1.0.3 #3 added two failure-observation surfaces
(WARNING log, `Runtime.errors`). A first-class
`run_goal(..., on_failure=callback)` parameter is a larger
affordance, deferred to v1.1.

Open design questions per CONTRACT v1.0.3 #3:
- Synchronous vs asynchronous callback shape?
- Whether the callback can suppress the `behavior.failed` event
  entirely (likely no — events are the audit trail).
- Whether `on_failure=` composes with explicit
  `@behavior(on=['behavior.failed'])` subscribers (they should
  coexist, but the ordering needs locking).

**Source**: CONTRACT v1.0.3 #3 "v1.1 candidate" + "v1.1 backlog
candidates."

**Dependencies**: none.

**Priority**: nice-to-have. The v1.0.3 #3 affordances already give
callers structured access; the callback is ergonomics.

**Scope**: medium. The callback wiring is small; the design pass
is where the work lives.

#### B-6 — Fire-once-on-condition aggregation triggers

**Description**: from second user-test report (finding E): LLM
behaviors subscribed to high-volume event streams want
"fire-once-when-conditions-are-met" semantics. The user worked
around with a "gate behavior" pattern — a Python `@behavior`
creates a control object exactly once when conditions are met; the
LLM behavior subscribes to the control object.

v1.1 scope: name the pattern, expose it as a first-class
decorator argument (or a `@trigger` helper), document the
gate-behavior pattern as the fallback when the trigger doesn't
fit.

**Source**: CONTRACT v1.0.3 §"v1.0.3 deliberately does NOT touch"
Finding E; restated in v1.1 backlog candidates #4.

**Dependencies**: none.

**Priority**: should-have. Real user friction surfaced this; the
workaround is documented but not ergonomic.

**Scope**: medium. The API shape design is the hard part; the
implementation is a layer over existing scheduling.

#### B-7 — `Runtime.load` auto-provider ergonomics

**Description**: rc1 user-test surfaced B2: `Runtime.load(...)`
crashed with `MissingProviderError` when the loaded diligence pack
had LLM behaviors and the caller didn't pass `llm_provider=`. The
rc2 tactical fix passes `RecordedDiligenceProvider` explicitly; v1.1
opens the design question of whether `Runtime.load` should infer
a provider from the run's recorded events, a pack-manifest
declaration, or a sidecar in run metadata.

Open design questions per CONTRACT v1.1 #5:
- `auto_provider=True` flag on `Runtime.load` (opt-in inference).
- Pack-declared default provider.
- Sidecar in run metadata.

Replay-strict interaction must lock alongside the choice.

**Source**: CONTRACT v1.1 #5; user-test B2 finding.

**Dependencies**: none.

**Priority**: should-have. Adoption-surface friction surfaced by a
real user test.

**Scope**: medium. The chosen shape determines scope; sidecar is
largest, `auto_provider=True` flag is smallest.

### Theme C — API surface coherence (from this review)

#### C-1 — Deprecate `graph.query` / `object_type=` kwarg

**Description**: v1.0.3 #1 added `graph.objects(type=, where=)` as
canonical; kept `graph.query(object_type=, where=)` as a
backward-compatible alias with explicit "no deprecation in v1.0.3"
carve-out. The kwarg name divergence (`type` vs `object_type`)
between canonical and alias is internally inconsistent.

v1.1 scope: emit `DeprecationWarning` on `graph.query` calls and
on `object_type=` kwarg use. Update remaining callers (internal
`tools/graph_query.py`, pack examples, docs).

**Source**: CONTRACT v1.0.3 #1 "Why not deprecate graph.query in
v1.0.3"; review §2 Finding C.

**Dependencies**: none.

**Priority**: should-have. The longer the alias lives, the harder
the eventual removal.

**Scope**: small.

#### C-2 — Full Graph/View surface harmonization

**Description**: review §2 Findings A, B, F. `graph.relations()`
absent (also a v1.0.4 candidate as Finding A); `graph.events` is
a property while `view.events(type=...)` is a method;
`graph.all_objects()` / `all_relations()` coexist with
`graph.objects()` / a hypothetical `graph.relations()`.

v1.1 scope: lock a Graph-side method shape that mirrors View
exactly for read access (`graph.objects`, `graph.relations`,
`graph.events` all callable with optional `type=`/`where=`),
deprecate the no-arg variants (`all_objects`, `all_relations`,
`events` property), preserve `get_object` / `get_relation` /
`get_relations` for endpoint-shaped queries.

**Source**: review §2 Findings A, B, F. Finding A is also v1.0.4
candidate v1.0.4 #1 — the smallest piece can ship sooner.

**Dependencies**: a deprecation cycle, so should land alongside C-1.

**Priority**: should-have.

**Scope**: medium. The API surface is small; the impact on
existing callers is medium.

### Theme H — Fork / cache semantics

#### H-1 — Fork cache pre-population symmetry

**Description**: `Runtime.fork(at_event=...)` in-process
pre-populates the LLM cache from the parent's recorded
`llm.responded` events (per CONTRACT v0.6 #8). The persistent
shape (`SQLiteEventStore.fork_run` → `Runtime.load(run_id=fork)`)
does NOT — the loaded fork sees only the fork's events, ending at
the fork point.

v1.1 scope: `Runtime.load` of a fork run should look up the
parent's `forked_at_event_id` (already on the `runs` row) and
pre-populate the cache from the parent's events through that
point. New runtime behavior, banned in v1.0.

**Source**: CONTRACT v1.1 #7-and-beyond.

**Dependencies**: none.

**Priority**: should-have. Asymmetry surfaced in rc2 #4/8 B2 fix;
fork workflows in production hit it.

**Scope**: medium.

## What v1.1 deliberately does NOT include

Stays consistent with the v1.0 deferral list (CONTRACT v1.0 #10):

- Web UI (never).
- Streaming LLM responses (post-1.0+).
- Multi-model routing / fallback (post-1.0+).
- More packs beyond Diligence (post-1.0+).
- Pack registry or marketplace.
- Error message i18n (never; English only).
- Video tutorials (separate project).
