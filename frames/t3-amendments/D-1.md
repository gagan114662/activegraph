# Frame t3 amendment D-1 — replay semantics

**Frame:** `t3-implement-cli-set-flag` (outer: `04a6dd5`)
**Author:** Spec Owner, 2026-05-21
**Decision class:** behavioral — locks user-visible semantics
**Status:** PROPOSED — awaiting Frame Architect review before D-2 begins

---

## Reframing (Atlas, 2026-05-21)

The yaml's D-1 asked: "does replay use the new value (0.7) or the value
that was active when the fork was originally created?" Atlas reframed
this to the more foundational question: **what does `replay()` do when
it encounters a `set` event on a key that was already set in a prior
event in the same log?**

Reframing accepted. Once the in-log replay semantics are locked, the
yaml's original question dissolves — see "Downstream resolution" below.

## Picks considered

| Pick | Behavior | CONTRACT alignment |
|---|---|---|
| last-write-wins (LWW) | later event clobbers earlier; replay state = final value | Trivially deterministic, but silently masks the earlier event's effect — soft violation of §3 (event immutability is about append-only, but the *semantic* of the earlier event is being erased at replay) |
| first-write-wins (FWW) | replay treats first set as authoritative; later sets are no-ops or errors | Same soft §3 issue, in the opposite direction; also breaks the natural reader expectation that later events refine state |
| **monotonic-only** | replay rejects log if a key is set twice with *different* values; same-value re-sets are idempotent | Matches §v0.5 #7's `ReplayDivergenceError` pattern exactly; honors §3 by reading both events and validating rather than masking either; honors §8 determinism (same log → same outcome, including same raise) |

## Pick

**monotonic-only with same-value idempotence.**

On replay, when the projector encounters a `set` event for key `K` with
value `V`, it consults the current override state:

- If `K` is not yet set → record `K=V`. (normal path)
- If `K` is already set to `V` → idempotent no-op. (same-value re-set;
  legal because §3 says corrections happen by new events, and a
  re-affirmation is a degenerate correction)
- If `K` is already set to `V' ≠ V` → raise
  `ReplayDivergenceError(event_id=<second set's id>, expected=V',
  actual=V)`. (conflicting set, exactly the pattern §v0.5 #7 locks)

## CONTRACT citations

- **§3 Event immutability** (CONTRACT.md:124–127) — "Events are
  append-only. Never edited, never deleted. Corrections happen by
  emitting new events." LWW/FWW silently mask one event's semantic;
  monotonic-only reads both and validates.

- **§8 Determinism** (CONTRACT.md:177–185) — "The runtime injects
  deterministic `clock` and `random` for replay." Same log MUST produce
  same outcome on every replay. Monotonic-only satisfies this: a log
  that raises raises every time; a log that converges converges every
  time.

- **§v0.5 #7 Determinism becomes load-bearing** (CONTRACT.md:449–456)
  — `replay_strict=True` raises `ReplayDivergenceError(event_id,
  expected, actual)` on first mismatch. Monotonic-only is the
  natural extension of this pattern from "event-type stream mismatch"
  to "override-key value mismatch."

- **§2 Event log is the source of truth (strict)** (CONTRACT.md:112–122)
  — the override value lives in the event, not in a projector cache
  that could shadow it; both LWW and FWW work by treating the projector
  as the canonical resolver, which is a structural inversion.

## Downstream resolution: yaml's original D-1

The yaml asked whether replay uses the fork-time override (0.7) or
"the value that was active when the fork was originally created."
Under monotonic-only, this question dissolves:

- The `--set` invocation at fork creation emits a `set` event into the
  new fork's log with `value=0.7`. That event is frozen by §3.
- Replay reproduces the override from the event (not from the pack
  default at replay time). The value at the event IS the value.
- If the pack's default later changes from 0.5 → 0.6 in code, the
  fork's recorded `--set 0.7` is unaffected: replay still produces 0.7
  because the log says 0.7.
- If a future runtime version mutates the pack default to a different
  type or removes the key, that's a separate failure (a
  schema-migration concern handled in D-3), not a replay semantics
  concern.

**The override is captured as an event-log snapshot at fork-creation
time. Replay reads the snapshot, not the pack default.**

## Cross-amendment ripple

This pick constrains the remaining amendments:

- **D-2 (bad-value handling):** the `set` event must contain a
  validated value before it lands in the log; otherwise replay can
  encounter a value that re-validates differently in a future runtime
  (drift across §v0.5 #7's payload-drift limitation). Tilts D-2
  toward fail-fast at CLI parse time (option a) or fork construction
  (option b), away from "store as-is and explode at replay" (option c).

- **D-3 (persistence schema):** the override must round-trip through
  SQLite identically; if stored as a JSON blob, the JSON encoding
  must be canonical (sorted keys, no trailing whitespace) to preserve
  the same-value idempotence check across reads. Tilts D-3 toward a
  schema where override key/value are first-class queryable fields
  rather than opaque blobs.

- **D-4 (type coercion):** the value stored in the event must be the
  coerced value, not the raw string. Otherwise `0.7` (parsed float)
  and `"0.7"` (raw string) would be treated as conflicting under
  monotonic-only. Tilts D-4 toward (b) infer types or (c) explicit
  typed form, away from (a) strings-only.

These are signals for the next three rounds, not pre-commitments —
Frame Architect and Spec Owner decide each D-N on its own merits.

## Test obligations created by this pick

- A test must demonstrate that replay of a log with two
  same-value `set` events for the same key is idempotent.
- A test must demonstrate that replay of a log with two
  different-value `set` events for the same key raises
  `ReplayDivergenceError` with the second event's id.
- A test must demonstrate that a fork-of-fork where parent and child
  both `--set` the same key to different values raises at fork
  creation time (because the child's `set` event would conflict with
  the parent's already-replayed `set` event during fork construction
  per §v0.5 #14 — `Runtime.fork` uses the replay path).

These will be anchored to predicate `replay_honors_set_override_per_D1_decision`
in the yaml.

## Rejected with reasoning

- **LWW:** simpler, but normalizes silent override of a prior event's
  effect. Pattern leaks into D-2/D-3 — if LWW is OK at replay, why
  not at runtime? Answer becomes "because of §3" — at which point
  monotonic-only is the cleaner draw of the line.

- **FWW:** equivalent §3 issue with opposite polarity, plus the
  ergonomic surprise that re-issuing `--set` against an already-set
  key silently does nothing. Worst of both worlds for user trust.

- **Implicit "fork-time pack default freeze" (not in Atlas's three):**
  rejected because it treats the pack registry as the canonical
  source instead of the event log, inverting §2.
