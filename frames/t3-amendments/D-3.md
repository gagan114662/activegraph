## Question

What does the `fork.override.applied` event payload contain, and how
is the override value serialized?

Options considered:

- **(a) Minimal:** `{key, value}` only; replay validates against the
  current pack schema.
- **(b) Pinned-version:** `{key, value, pack_version, value_type}`;
  replay validates against the pinned pack version.
- **(c) Full attestation:** `{key, value, pack_version, value_type,
  schema_constraint_snapshot}`; replay is self-sufficient and does not
  need the pack registry to explain or validate the historic write.

## Pick

**(c) Full attestation, with JSON-native typed value serialization.**

Commit subject: `Frame t3 amendment D-3: full-attestation — schema snapshot makes fork.override.applied replay self-sufficient across pack drift`

Persist `fork.override.applied` as an event whose payload is small but
self-contained:

```json
{
  "schema_version": 1,
  "pack": {
    "name": "diligence",
    "version": "0.1.0"
  },
  "setting": {
    "key": "confidence_threshold_for_review",
    "value": 0.7,
    "value_type": "float"
  },
  "schema_constraint_snapshot": {
    "source": "Pack.settings_schema",
    "model": "DiligenceSettings",
    "field": "confidence_threshold_for_review",
    "json_schema": {
      "type": "number",
      "minimum": 0,
      "maximum": 1
    },
    "constraint_hash": "sha256:<canonical-field-schema-json>"
  }
}
```

The value is serialized in JSON-native typed form: floats and ints are
JSON numbers, bools are JSON booleans, strings are JSON strings, lists
and objects stay JSON arrays/objects if the settings schema permits
them. `value_type` records the canonical type D-4's coercion boundary
produced before D-2 validation succeeded.

This is not the tagged-union form (`{"type": "float", "val": 0.7}`)
because `value_type` already tags the JSON value. It is not the
always-string form because stringifying would erase the same-value
idempotence D-1 needs (`0.7` and `"0.7"` must not compare equal by
accident).

## Replay semantics

Replay trusts the event payload as the historic truth:

1. It validates the payload shape against `schema_version == 1`.
2. It applies the recorded `setting.value` as the effective override
   from the fork point forward.
3. It does not require the current pack registry, the current
   `Pack.settings_schema`, or the old pack version to be importable.
4. It may use `schema_constraint_snapshot` for corruption diagnostics
   and human inspection, but not as permission to reinterpret the
   value through today's schema.

If the current pack later tightens a range from `[0,2]` to `[0,1]`,
a historic fork with `temperature=1.5` still replays cleanly because
the event attests that `1.5` was valid under the constraint snapshot
that existed at fork creation. New forks created after the tightening
must pass D-2 validation against the new schema and would be rejected.

## Justification

### Replay self-sufficiency

Full attestation is the only option that survives all registry-drift
failures:

- pack registry unreachable
- old pack version deleted
- old pack module refactored or renamed
- current `Pack.settings_schema` tightened, loosened, or rewritten

D-1 says replay reads from the log, not ambient state. D-2 says the
validator's successful write is the attestation point. D-3 therefore
persists the attestation itself so replay does not reconstruct the
past from a mutable registry.

### Constraint evolution

Constraint evolution is write-time scoped:

- Historic event: replays under its recorded
  `schema_constraint_snapshot`.
- New event: validates against the currently loaded
  `Pack.settings_schema` before it is written.

That preserves both rules operators care about: old valid forks do
not rot when a pack tightens constraints, and new forks cannot bypass
the tightened constraints by citing an older event.

### Payload size budget

The snapshot is intentionally field-scoped, not model-scoped. Persist
only the selected field's Pydantic JSON schema fragment plus a hash of
its canonical form. For normal scalar settings this is tens to a few
hundred bytes. That cost is worth paying because it removes a hidden
runtime dependency on old pack code and makes `activegraph inspect
--event <evt>` explain why the value was valid when written.

The implementation must not persist the entire pack manifest or full
model schema unless a nested setting genuinely requires it. Start with
the smallest field-level `model_json_schema()` fragment that can
explain type, bounds, enum, nullable, and default semantics for the
chosen key.

## Rejected with reasoning

### (a) Minimal `{key, value}`

Rejects the log-is-truth rule in practice. Replay would need today's
pack schema to decide whether yesterday's value was valid. If the
pack tightens `[0,2]` to `[0,1]`, an old fork with `1.5` becomes
unreplayable even though the event was valid when written.

That makes replay correctness depend on a mutable out-of-log surface.
It contradicts CONTRACT §2 and D-1's lock-at-creation rule.

### (b) Pinned-version `{key, value, pack_version, value_type}`

Better than (a), but only if the team commits to a permanent pack
version registry. No such registry is part of the current contract for
CLI replay, and adding one would turn `fork --set` into a pack
archival feature. If old pack versions are missing, replay still
cannot explain or validate the event.

(b) also leaves the operator with a version reference but no local
answer to "what constraint accepted this value?" Full attestation
answers that question from the event alone.

### Tagged-union value serialization

Rejected as redundant with `value_type`. A tagged union makes every
reader unwrap `setting.value.val` and invites two sources of truth
(`setting.value.type` vs `setting.value_type`). The value should be
the value.

### Always-string value serialization

Rejected because it erases type at the storage boundary. D-1's
same-value idempotence check and D-2's structured validation both
need canonical typed values. Persisting `"0.7"` would force replay to
run D-4 coercion again, recreating the exact ambient interpretation
dependency D-3 is avoiding.

## CONTRACT anchors

- **§2 Event log is source of truth** (CONTRACT.md:112-122) — the
  event carries the data needed to reconstruct state; no out-of-log
  pack registry is required for normal replay.
- **§3 Event immutability** (CONTRACT.md:124-127) — the payload is
  append-only; D-3 must get the bytes right at write time.
- **§v0.5 #11 Fork copies event rows** (CONTRACT.md:513-516) — rows
  under the new `run_id` are authoritative from landing.
- **§v0.5 #12 IDs are scoped to run_id** (CONTRACT.md:518-523) — the
  override event's `evt_NNN` identity is permanent once minted.
- **§v0.9 #7 Pack-scoped settings** (CONTRACT.md:1924-1960) —
  `Pack.settings_schema` is the stable validation source at write
  time; D-3 records the specific field constraint snapshot derived
  from that surface.

## Implementation notes for Code Owner

- Add a payload schema validator for `fork.override.applied` v1. It
  checks structure and JSON-serializability; it does not import the
  historic pack.
- Canonicalize `schema_constraint_snapshot.json_schema` before hashing
  (sorted keys, no insignificant whitespace). Store the hash for
  diagnostics and corruption detection.
- If Pydantic emits non-JSON-native defaults or metadata, normalize to
  a JSON-safe subset before persistence. The event log must remain
  ordinary JSON.
- D-4 owns how CLI strings become typed Python values. D-3 only says
  the post-coercion value is persisted in JSON-native typed form with
  `value_type`.
- Pre-D-3 forks have no `fork.override.applied` event and need no
  migration. Store schema migration is required only if the store
  needs an index over override events; the event payload itself can
  live in the existing event payload column.

## Test obligations created by this pick

- A fork created with `--set diligence.threshold=0.7` records a
  `fork.override.applied` payload containing `schema_version`,
  `pack.name`, `pack.version`, `setting.key`, JSON-native
  `setting.value`, `setting.value_type`, and a field-scoped
  `schema_constraint_snapshot`.
- Replay succeeds when the pack registry is unavailable, as long as
  the stored event payload is well-formed.
- Replay of an old fork still succeeds after the current pack schema
  tightens a range that would reject the old value for new forks.
- New fork creation after a schema tightening rejects the same value
  through D-2 validation before writing an event.
- A corrupted payload with mismatched `value_type` vs JSON value shape
  raises a payload-corruption error during replay/inspection.
