## Question (verbatim from yaml line 75–82)

Type coercion — `--set foo=bar` passes a string. The pack key
`diligence.threshold` is a float in code. Options: (a) strings only,
consuming code must coerce; (b) infer int/float/bool from the value
text; (c) require an explicit typed form like `--set foo:float=0.7`.
Pick one and name the failure mode for each rejected alternative.

## Pick

**(b) — Schema-driven coercion through the pack's existing
`Pack.settings_schema` Pydantic model. The CLI passes the raw string;
`validate_override(key, value, pack)` (D-2's helper) feeds the value
through Pydantic's per-field validator for `key`; the typed value
returned by Pydantic is what lands in `fork.override.applied.value`
per D-3's full-attestation payload.**

Commit subject: `Frame t3 amendment D-4: pydantic-schema — coerce override values through Pack.settings_schema field validators at validate_override; raw string in CLI, typed value in event log`

### Why this is (b) and not (c)-dressed-up

This is option (b) — inference from value text — but the inference is
**not ad hoc**. It is delegated to Pydantic's per-field validator,
already required to exist on every pack via `Pack.settings_schema`
(D-2 §"Pack-schema lookup surface"). The CLI does not guess; the
schema decides. A field declared `threshold: float` accepts `"0.7"`
and yields `0.7`. A field declared `name: str` accepts `"foo"` and
yields `"foo"`. A field declared `tags: list[str]` accepts a JSON
literal or fails per Pydantic's standard parsing rules.

The user-visible surface is exactly the form the docs already
promised across 8 markdown files: `--set <pack>.<key>=<value>`. No
typed-suffix syntax. No CLI-side type-inference rules to learn
separately from the pack's own schema.

## How coercion threads through D-1/D-2/D-3

```
   CLI                         validate_override                       event log
   ───                         ─────────────────                       ─────────
   "0.7"  ────raw string──▶    Pack.settings_schema field validator
                                 │
                                 ├── valid   ──▶  typed value  ──▶    fork.override.applied
                                 │                (e.g. 0.7)            .value = 0.7
                                 │                                      .value_type = "float"
                                 │                                      .schema_constraint_snapshot = {...}
                                 │                                      (D-3 full attestation)
                                 │
                                 └── invalid ──▶  InvalidOverrideError(
                                                     key, raw_value,
                                                     schema_constraint)
                                                  (D-2 reject pre-event)
```

D-1 invariants hold: the override is captured at fork-creation as a
typed value in the event payload; replay reads the typed value from
the log, never re-parses the CLI string. The CLI string is only the
ingress; once Pydantic has spoken, the typed value is canonical.

D-2 invariants hold: validation happens at fork-creation pre-event;
no `fork.override.applied` event is minted for a value the pack
schema rejects. The helper signature D-2 picked — `validate_override(
key, value, pack) -> None` — is the natural call site; D-4 only
specifies that the helper internally coerces via the field validator
and that callers retrieve the typed value from the helper's return
(see "Helper signature refinement" below).

D-3 invariants hold: the value persisted in the event payload is the
post-coercion typed value (JSON-native: number, string, bool, list,
dict). The `schema_constraint_snapshot` makes the payload
self-sufficient even if the pack's schema changes between fork and
replay.

### Helper signature refinement (binding, not optional)

D-2 declared `validate_override(key, value, pack) -> None` with a
return type of `None`. D-4 requires the typed value back, so the
signature must be refined to:

```python
def validate_override(
    key: str,
    raw_value: str,
    pack: Pack,
) -> CoercedOverride:
    """Validate and coerce a CLI override value against the pack schema.

    Returns a CoercedOverride record carrying (typed_value, value_type,
    schema_constraint_snapshot) for the caller to mint into the
    fork.override.applied event per D-3.

    Raises:
        InvalidOverrideError: if `key` is not in the pack's settings
            schema, or `raw_value` fails the field's Pydantic validator,
            or the validator coerces to a value outside any declared
            constraint (per D-2).
    """
```

This is a strict superset of D-2's signature: same call site, same
exception class, same pre-event timing — only the return shape grows
from `None` to a structured `CoercedOverride` record. D-2's
"unit-testable in isolation" property is preserved.

`CoercedOverride` is a thin dataclass; not a public-API addition.
It is consumed only by the fork-creation function that mints the
event. It exists so the typed value, the JSON-native value type, and
the schema constraint snapshot are computed in one place and handed
to the event minter as a single record — preventing the three pieces
from diverging across implementation iterations.

## Coercion edge cases

- **Boolean strings:** use Pydantic's field validator, not a separate
  CLI allow-list. Whatever the declared bool field accepts under the
  project's pinned Pydantic version is the coercion contract. Tests
  must cover lowercase `"true"` at minimum; adding `"1"`/`"0"`/`"yes"`
  coverage is allowed only if the pinned validator accepts them.
- **Integer strings:** use the declared pack field type. A field
  declared `int` must not be widened by a CLI-side float parser; any
  `"1.0"` behavior follows Pydantic's int-field validator and is
  surfaced as either a typed int or `InvalidOverrideError`, never as a
  hand-rolled truncation.
- **Empty strings:** `--set key=""` is the empty string. It is not
  `None`, not `--unset`, and not a tombstone. It succeeds only for a
  schema field that accepts the empty string; otherwise the helper
  raises `InvalidOverrideError` before any event is minted.
- **Unsupported/ambiguous shapes:** unions, optional values, enums,
  lists, and dicts are resolved by the pack schema. Enums require the
  exact value Pydantic accepts. `Optional[X]` does not introduce an
  unset surface; a future `--unset` command would be a separate frame.
- **Error payload:** coercion failures use
  `InvalidOverrideError(code="type_coercion_failure", key, value,
  expected_type, schema_constraint, cause)`, where `cause` carries the
  structured Pydantic `ValidationError` details. The error remains
  runtime-only because no fork and no event exist on failure.
- **`value_type`:** the event records the post-coercion,
  pack-schema-declared JSON-native type. It is not user-declared and
  not inferred independently from the raw CLI string.

## Rejected: (a) strings only, consuming code must coerce

Three failure modes:

1. **Replay-divergence by silent type drift.** D-3's `value_type` is
   the durable record of what was *coerced to* at fork time. If the
   CLI hands a raw string into the event log, `value_type = "str"`
   for every override — and the actual type the consuming pack
   coerces to becomes a runtime property of the consumer's coercion
   logic at replay time. The consumer can change between fork and
   replay (consumer is upgraded, regex changes, `int()` becomes
   `Decimal`, etc.) and the event log cannot distinguish the
   resulting divergence from a legitimate consumer change. This is
   precisely the v0.5 #7 `ReplayDivergenceError` class the D-1
   monotonic-only invariant exists to surface — but (a) makes the
   divergence undetectable because the log has no typed reference
   point to diverge from.

2. **Loss of D-2 fail-loud-at-fork-creation.** Pydantic field
   validators are where Pack authors declare type rules.
   `validate_override` cannot honor those rules without invoking the
   validator, which is exactly the coercion path. (a) forces either
   (i) duplicating Pydantic's validation logic in
   `validate_override` (drift hazard between two coercion
   implementations) or (ii) deferring all type errors to consumer
   code (failing the D-2 pre-event invariant).

3. **Replay-strict cookbook breaks.** Cookbook example at
   `docs/cookbook/common-patterns.md` shows the user setting
   `diligence.confidence_threshold_for_review=0.9` and expecting
   numeric semantics on replay. With (a), every consumer is
   responsible for `float("0.9")` — including the diff renderer, the
   replay validator, and any future consumer of the override value.
   Three sites instead of one. Drift across them is inevitable.

## Rejected: (c) explicit typed form `--set foo:float=0.7`

Three failure modes:

1. **Documentation lie surface area expands.** GAUNTLET T3 line 52
   notes that `--set <pack>.<key>=<value>` is the form documented in 8
   markdown files (CHANGELOG, CONTRACT, quickstart, common-patterns,
   migration, operating-in-production, concepts/forking,
   reference/cli). Picking (c) would either (i) require rewriting all
   8 files with the typed-suffix syntax, expanding the doc-vs-impl
   drift surface T2 just closed, or (ii) ship a CLI form the docs
   never advertised, which is the exact failure mode T2's gate
   exists to catch. Either path is a regression on T2's invariant.

2. **User must duplicate the schema in their head.** The pack schema
   already declares `threshold: float`. (c) forces the user to repeat
   `:float` at the CLI. If the user types `:int` for a `float` field
   the gate must reject — at which point the schema is still
   authoritative and the typed suffix was redundant friction. The
   user has been forced to learn a second type system.

3. **Composition with future flags becomes load-bearing.** Inspect
   uses `--memo` / `--search`. If overrides later need to support
   list types or nested dicts, (c)'s grammar must grow
   (`--set foo:list[str]=a,b,c`?). Each grammar extension is a
   user-visible breaking change. Pydantic already handles
   nested/list/dict; deferring to it side-steps grammar growth
   entirely.

## Boundary-anchored tests (per §7 #2 of CONTRACT-review-findings.md)

Anchor on the helper signature, not the CLI surface:

| Test | Anchor |
|---|---|
| `test_validate_override_coerces_int_to_float` | numeric widening through Pydantic field validator |
| `test_validate_override_coerces_int_to_int` | exact-type case |
| `test_validate_override_coerces_lowercase_true_to_bool` | Pydantic's standard bool parsing |
| `test_validate_override_rejects_alpha_for_float` | `InvalidOverrideError`, no event minted (D-2 invariant) |
| `test_validate_override_rejects_unknown_key` | `InvalidOverrideError`, schema lookup failure |
| `test_validate_override_returns_typed_value_for_event_payload` | typed value matches Pydantic's `model_dump` shape |
| `test_validate_override_returns_value_type_jsonable` | `value_type` is JSON-native string per D-3 |
| `test_validate_override_returns_schema_constraint_snapshot` | snapshot non-empty when field has constraints |

These anchor on the helper's contract, not on the click subcommand —
which means CLI refactors (typer adoption, subcommand reshuffling)
do not destabilize the type-coercion test suite. Same discipline as
T2's D-1 static extraction anchoring on AST shape, not surface form.

## Cross-amendment ripple

- **D-1 (lock-at-creation, monotonic):** unaffected. The typed value
  is still captured once at fork-creation; same-value re-sets are
  still idempotent at the typed-value comparison (not the raw-string
  comparison — `--set t=0.7` followed by `--set t=0.70` must
  resolve to the same idempotent re-set, which only works because
  Pydantic coerces both to `0.7`).
- **D-2 (reject pre-event, helper):** signature refined from
  `-> None` to `-> CoercedOverride`. Same call site, same exception
  class, same timing.
- **D-3 (full attestation):** `value`, `value_type`, and
  `schema_constraint_snapshot` are all sourced from `CoercedOverride`
  returned by the helper. The event minter does not re-parse, does
  not re-introspect the schema.

## Future framework caveat

If pack settings ever need to accept arbitrary Python objects
(non-JSON-serializable), the D-3 full-attestation payload cannot
serialize them — and D-4's Pydantic coercion would have to be
sidestepped via `model_config["arbitrary_types_allowed"] = True`.
That's a CONTRACT v1.2 candidate, not a T3 concern. The current
diligence pack's settings are all JSON-native; D-4 ships against
that corpus.
