## Question (verbatim from yaml line 54–63)

Bad-value handling — if `--set diligence.threshold=not_a_number` is
passed, when does the failure surface? Options: (a) fail-fast at CLI
parse time before fork creation; (b) fail at fork construction when
the override would be applied to the pack default; (c) store as-is
and explode at replay when the consuming code type-coerces.

## Pick

**(b.ii) — Reject at fork-creation pre-event, via a dedicated
`validate_override(key, value, pack) -> None` helper that raises
`InvalidOverrideError(key, value, schema_constraint)` on failure.**

Commit subject: `Frame t3 amendment D-2: reject at fork-creation pre-event — validate_override(key, value, pack) helper raises InvalidOverrideError before fork.override.applied is minted`

### Sharpening between (b.i) inline vs (b.ii) helper

(b.i) inlines the pack-schema lookup + validation directly inside the
fork-creation function. Same semantics, but the validator is not
unit-testable in isolation and is not reusable by future override-mint
paths.

(b.ii) factors the validation into a dedicated
`validate_override(key, value, pack) -> None` helper. Same call site
inside fork-creation, same pre-event timing — but the helper is
unit-testable in isolation, reusable by any future override-mint path
(D-3 persistence/restore and D-4 type-coercion both want to call it),
and the boundary-anchored test obligation from CONTRACT-review-findings
§7 #2 maps to the helper directly rather than to a fork-creation
integration test.

**(b.ii) chosen** — D-3 and D-4 both create new call sites that will
want this validator. Code Owner factors once, calls thrice.

### Pack-schema lookup surface

D-2 does **not** introduce a new generic pack-introspection API. The
stable surface for override validation is the existing
`Pack.settings_schema` Pydantic model, already required by
`activegraph.packs.Pack`.

`validate_override(key, value, pack)` reads the pack parameter schema
from `pack.settings_schema` (`model_fields` / `model_json_schema()` as
needed by the implementation), not from ad hoc CLI metadata. D-3's
persisted event payload therefore references the pack identity
(`pack.name`, `pack.version`) plus the override key/value; it does not
persist a separate schema object.

If Code Owner discovers a nested-key shape that cannot be represented
cleanly against `Pack.settings_schema`, the smallest allowed new API is
a local helper such as `override_schema_for_pack(pack)`, implemented
as an adapter over `pack.settings_schema`. It is not a new public
`pack.schema()` contract unless D-3 explicitly reopens that boundary.

### Error payload completeness

`schema_constraint` is a structured runtime object, not a human-only
string and not a persisted artifact. It must carry at least:

- `code`: `unknown_key`, `type_mismatch`, or `constraint_violation`
- `expected_type`: the declared type when applicable
- `bounds` / `enum`: the violated constraint when applicable
- `message`: a renderable CLI-facing explanation derived from the
  structured fields

The error is runtime-only because fork creation failed: no fork exists,
no `fork.override.applied` event exists, and there is no event log
surface to persist the error into. D-3 should persist only valid,
already-validated override payloads.

## Behavior

```
fork creation flow:
  CLI parses `--set <key>=<value>` → raw (str, str) tuple
  ↓
  fork-creation function:
    resolve pack manifest for the target run/fork
    validate_override(key, value, pack)         # ← raises pre-event
    mint fork.override.applied event            # ← only on validation pass
    persist event under new run_id
```

`validate_override(key, value, pack)` checks:

1. **Key exists in pack schema** → else `InvalidOverrideError(key,
   value, "unknown_key")`. Catches `--set typo.threshold=0.7`.
2. **Value coerces to declared type** → applies D-4's coercion rule
   (D-4 is downstream; D-2 only locks the call site, not the coercion
   policy). Else `InvalidOverrideError(key, value, "type_mismatch:expected=<T>,got=<observed>")`.
3. **Value satisfies declared range/enum constraint if present** →
   else `InvalidOverrideError(key, value, "constraint_violation:<expr>")`.
4. **Helper raises pre-event.** No `fork.override.applied` is minted
   if any check fails. The fork is not created — caller sees the
   exception, no partial state, no orphan event.

## Justification

### CONTRACT anchors

- **§v0.5 #11 Fork copies event rows** (CONTRACT.md:513–516) — fork
  creation copies parent rows under a new `run_id`, then appends the
  fork's own seed events (including `fork.override.applied`).
  Everything written under the new `run_id` is part of the fork's
  authoritative log from the moment it lands.

- **§v0.5 #12 IDs are scoped to run_id** (CONTRACT.md:518–523) — the
  fork's `evt_NNN` counter is reseeded at creation; once a `fork.override.applied`
  event takes an evt_id, that id is permanent under the run_id.
  Writing a known-invalid value would burn an evt_id on a row replay
  must reject — a self-inflicted divergence.

- **§3 Event immutability** (CONTRACT.md:124–127) — "Events are
  append-only. Never edited, never deleted. Corrections happen by
  emitting new events." A bad event cannot be retroactively fixed.
  The only way to keep the log clean is to refuse to write the bad
  event in the first place.

- **§2 Event log is source of truth (strict)** (CONTRACT.md:112–122) —
  "Every state change is an event first, projection second." There is
  no out-of-band validation cache that replay can consult. The event
  itself must already be valid, because replay reads only the log.

- **D-1 LOCKED** (inner: `27d488b`) — producer rule says replay reads
  the override value from the event payload (lock-at-creation, `316a052`);
  log invariant says conflicting same-key set events raise
  `ReplayDivergenceError` (`eb9d154`). If replay's source of truth is
  the event payload, that payload MUST be valid at write time —
  otherwise (c)'s "explode at replay" path becomes a guaranteed-future
  failure baked into the log, indistinguishable at write time from a
  legitimately-stored override.

- **Boundary-anchored test obligation** (CONTRACT-review-findings.md §7 #2)
  — the D-2 predicate (`cli_rejects_malformed_set_per_D2_decision`) must
  anchor on the validation moment (where the error surfaces), not on
  the implementation path (which function calls which). Picking a
  dedicated `validate_override` helper makes the anchor location
  unambiguous: tests assert against the helper's behavior in isolation
  AND against the fork-creation integration path's refusal to mint
  the event when the helper raises.

### Why (b.ii) is forced under D-1

Under D-1's lock-at-creation rule (producer), the event payload IS
the truth replay reads. Under D-1's monotonic-only invariant (log),
replay refuses to silently mask divergent same-key events. Both
require the event to be valid when written:

- If the event payload encodes a type-invalid value (e.g.
  `temperature="not_a_number"` against a float-declared key), replay
  reads it back as a string and the consuming code's type-coercion
  blows up — the failure is deferred, not prevented.
- If the event payload encodes a range-violating value (e.g.
  `temperature=99.9` against a `[0,2]` constraint), replay reads it
  back faithfully and the consuming code is given a value outside
  its contract — the failure is deferred, not prevented.

D-1 said the log is the truth. (c) says the log can lie until first
read. These are inconsistent. (b.ii) is the smallest factoring that
keeps the log honest from the moment of write.

## Rejected with reasoning

### (a) Fail-fast at CLI parse time

Catches **syntactic** failures only — `--set foo=bar` accepted because
it parses as `(str, str)`, even if `foo` isn't a key in the target
pack or `bar` doesn't coerce to the declared type. The pack schema is
not available at the argparse layer (argparse runs before pack
resolution).

Failure mode: `--set diligence.threshold=99.9` (syntactically valid,
semantically invalid against a `[0,2]` range constraint) gets through
the CLI and lands as a `fork.override.applied` event. The bad value
sits in the log until the consuming behavior tries to read it,
violating "fork creation succeeds → log is replayable."

(a) is necessary but not sufficient — basic CLI syntax errors (e.g.
`--set` with no `=`) should still raise at parse time. But the
semantic validation (key existence, type, constraints) MUST happen
at fork-creation, post-pack-resolution. So (a) collapses into a thin
syntactic check that's a prerequisite for (b.ii), not a replacement.

### (c) Store as-is, explode at replay

Direct contradiction with D-1's lock-at-creation rule. Under D-1, the
event payload is the truth replay reads. Storing a known-bad value
poisons the log permanently — §3 immutability means we can't repair
the event, only emit corrections, but there's no semantic for
"correction of a malformed override" in CONTRACT v0.5 or v1.0.

(c) also breaks the predicate
`replay_in_fresh_venv_against_pre_migration_db_passes` — a fresh-venv
replay against a db with a known-bad override event would fail
deterministically, with no path to recovery short of editing the
log (which §3 forbids).

(c) is rejected categorically, not just unfavored. Code Owner must
not implement this path.

### (b.i) Inline validation without helper

Same semantics as (b.ii), but couples the validation logic to the
fork-creation function's internal structure. Three concrete costs:

1. **No isolation test.** A boundary-anchored test for D-2 (per
   CONTRACT-review-findings §7 #2) wants to assert against the
   validation moment without spinning up a fork-creation integration.
   (b.i) forces integration tests for every validation case.

2. **No reuse for D-3/D-4.** D-3 (persistence schema) will have a
   restore-from-disk path that re-reads override events and validates
   them against the current pack schema (catches the
   "schema-changed-since-write" case). D-4 (type coercion) will need
   the validator at the coerce-then-store boundary. (b.i) means
   re-implementing the validator at each call site.

3. **Refactor cost.** D-3 or D-4 picks that surface additional call
   sites force a (b.i) → (b.ii) refactor under time pressure during
   Code Owner's impl phase. Doing the factoring now is cheap (one
   helper, three call sites at most); doing it under D-3 pressure
   risks subtle drift between call sites.

The semantics are identical; (b.ii) wins on testability and
reusability. No semantic ground given up.

## Cross-amendment ripple (signals only, not pre-commitments for D-3/D-4)

- **D-3 (persistence schema):** the restore path will re-hydrate
  override events from the store. If the pack schema has evolved
  since the event was written (e.g. a key's type changed from int to
  float), `validate_override(key, stored_value, current_pack)` may
  now raise. D-3 must pick whether restore tolerates schema drift
  (and how) — but the validator helper from D-2 is the natural
  attestation point. Tilts D-3 toward an explicit schema-version
  field on the fork record, queryable independently of the override
  payload.

- **D-4 (type coercion):** the helper's "value coerces to declared
  type" check requires D-4's coercion rule. D-4 picks the coercion
  policy (strings only / infer / explicit typed form). D-2 only locks
  WHERE the coercion validation happens (inside `validate_override`,
  pre-event), not WHAT the coercion policy is. Tilts D-4 toward an
  explicit typed form (option c) — the validator's error message is
  cleanest when the user-declared intent (`--set foo:float=0.7`) is
  available, and the validator can refuse ambiguity rather than
  silently inferring.

## Test obligations created by this pick

Per CONTRACT-review-findings §7 #2, the boundary-anchored test for
D-2 anchors on **the validator helper's behavior**, not on the
fork-creation integration path. Required tests:

- `validate_override` raises `InvalidOverrideError(key, value, "unknown_key")`
  on a key not in the pack schema.
- `validate_override` raises `InvalidOverrideError(key, value, "type_mismatch:...")`
  on a value that fails D-4's coercion rule.
- `validate_override` raises `InvalidOverrideError(key, value, "constraint_violation:...")`
  on a value that fails a declared range/enum constraint.
- `validate_override` returns `None` (no raise) on a valid (key,
  value) pair against the pack schema.
- Fork-creation integration test: when `validate_override` raises,
  no `fork.override.applied` event is minted, the fork is not
  created, and the caller sees the `InvalidOverrideError` unchanged
  (not wrapped).
- Replay integration test: a fork created with a valid override
  replays without re-running `validate_override` on the recorded
  event (the event was validated at write time; replay trusts the
  log). Documents the "validate at write, trust at read" contract.

These anchor to predicate `cli_rejects_malformed_set_per_D2_decision`
in the yaml, plus the new predicate `boundary_anchored_test_for_D2_validation_moment`.

## Open items for D-3

D-3 must pick the persistence schema for fork records with overrides
(new column / JSON blob / sidecar table) and define the migration
path for pre-D-2 fork records (which by construction have no
overrides). D-2's `validate_override` helper does not constrain the
schema choice — only the round-trip property: whatever is read out
of storage must, when re-passed through `validate_override` against
the (possibly evolved) pack, raise iff it would have raised at the
original write time. Tilts D-3 toward an explicit schema-version
field, but does not pre-commit.
