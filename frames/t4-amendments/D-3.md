## Question

How will the same tool-using behavior run against AnthropicProvider and
OpenAIProvider without live network, and what event sequence must be identical?

## Pick

Use fake SDK clients at the provider boundary, not live network and not a
runtime-only scripted provider.

The parity test runs the same `@llm_behavior`, same `@tool`, same freshly
constructed initial graph fixture, same `Runtime` budget, and same output schema
twice:

1. `AnthropicProvider(client=fake_anthropic_client)`
2. `OpenAIProvider(client=fake_openai_client)`

The fakes return provider-native wire objects:

- Anthropic fake: `content=[text block, tool_use block]`, then final text JSON.
- OpenAI fake: `choices[0].message.tool_calls`, then final text JSON.

This proves the provider adapters, runtime tool loop, tool events, and final
handler path together. A runtime-only `ScriptedProvider` is not enough for T4
because it bypasses the exact OpenAI shape this frame exists to translate.

Commit subject: `Frame t4 amendment D-3: fake-provider-parity — same behavior, native provider fixtures, normalized event-sequence match`

## Clarification after Sasha challenge (`inner:329c2b4`)

Sasha's D-3 challenge is accepted as blocking clarification. The parity fixture
strategy remains fake provider-native SDK clients, but the fixture construction
and comparison surface are now tighter.

### Fresh graph fixtures, not one reused graph

"Same Graph" means two fresh `Graph` instances created by the same deterministic
fixture factory, not the same mutable graph object reused across provider runs.
The factory must use the same initial objects, behavior registration, clock/id
seed if applicable, and runtime budget for both provider runs. The test must
assert the two graphs have identical seed state before either provider run
begins.

This prevents cross-run contamination from Anthropic events, queued behavior
state, or already-created objects leaking into the OpenAI run.

### OpenAI fake records both SDK calls

The fake OpenAI client must record every SDK call. The happy-path parity fixture
must assert there are two OpenAI calls and that call 2's `messages` contain:

1. an assistant message with OpenAI-native `tool_calls`; and
2. a following `role="tool"` message whose `tool_call_id` matches that assistant
   tool call id.

Event parity alone is insufficient because a canned fake could return the right
responses while the provider sends an invalid second-turn OpenAI request.

### Failure fixtures compare partial streams

Failure fixtures compare normalized partial event type sequences plus terminal
reason. Terminal reason parity alone is too weak.

Required partial sequences:

```text
invalid args:
goal.created
llm.requested
llm.responded
tool.requested
tool.responded
behavior.failed

unknown tool:
goal.created
llm.requested
llm.responded
behavior.failed

final parse failure:
goal.created
llm.requested
llm.responded
tool.requested
tool.responded
llm.requested
llm.responded
behavior.failed
```

The expected terminal reasons remain `tool.invalid_input`,
`tool.unknown_tool`, and `llm.parse_error`, respectively.

## Event sequence to compare

Compare the normalized event stream from both runs after applying the T4 parity
event filter. The filter includes:

- `goal.created`
- `llm.*`
- `tool.*`
- final user-domain `object.created`
- `behavior.failed` for failure fixtures

The filter excludes runtime lifecycle events such as behavior start/complete
events unless the implementation elects to include lifecycle events in both
provider streams consistently. Theo should use the filter above as the default
comparison surface.

Happy path normalized sequence:

```text
goal.created
llm.requested
llm.responded
tool.requested
tool.responded
llm.requested
llm.responded
object.created
```

The comparison must include these payload fields:

- event `type`
- `behavior` on `llm.*` and `tool.*`
- `tool` name
- canonicalized tool `args`
- `args_hash`
- `tool.responded.error`
- final object type and data
- `behavior.failed.reason` if any failure path is being tested

Provider-specific fields may be normalized away:

- model name
- token counts
- cost
- latency
- raw provider finish reason spelling (`"tool_use"` vs `"tool_calls"`)

For the happy path, both provider fixtures should use the same tool call id
(`"call_1"`) so `tool.requested.call_id` and the following tool-result echo can
also match exactly.

## Required fixtures

Minimum fixture matrix:

| Fixture | Anthropic shape | OpenAI shape | Expected result |
|---|---|---|---|
| happy path | `tool_use` block | `message.tool_calls` | identical normalized stream |
| invalid args | bad/missing required field | invalid/missing arguments JSON | `tool.invalid_input` |

| final parse failure | final text not JSON | final text not JSON | `llm.parse_error` |

## Non-goals

- Do not hit live Anthropic or OpenAI.
- Do not compare provider wire payloads byte-for-byte.
- Do not require identical token/cost accounting between providers.
- Do not change runtime call sites to make parity easier.
