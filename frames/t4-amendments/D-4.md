## Question

OpenAI currently raises `llm.network_error` for tools. Name the allowed reason
codes after implementation and confirm no new reason code is introduced unless
CONTRACT v0.6 #11 is amended.

## Pick

No new reason codes. The current OpenAI "tools unsupported" pre-call
`llm.network_error` goes away because tools become supported.

Allowed reason codes after T4 are the existing LLM, tool, and budget codes:

### LLM/provider side

- `llm.network_error`
- `llm.rate_limited`
- `llm.parse_error`
- `llm.schema_violation`
- `llm.fixture_missing`
- `llm.prompt_assembly_error`
- `budget.cost_exhausted`

### Runtime/tool side

- `tool.timeout`
- `tool.network_error`
- `tool.invalid_input`
- `tool.invalid_output`
- `tool.execution_error`
- `tool.unknown_tool`
- `tool.fixture_missing`
- `tool.max_turns_exhausted`
- `budget.tool_calls_exhausted`
- `budget.cost_exhausted`

## Mapping table

| Failure | Reason |
|---|---|
| OpenAI SDK timeout/connection/auth-shaped failure | `llm.network_error` |
| OpenAI 429-shaped failure | `llm.rate_limited` |
| Final response has no parseable JSON | `llm.parse_error` |
| Final response JSON violates output schema | `llm.schema_violation` |
| Tool call arguments fail declared input schema | `tool.invalid_input` |
| Tool name is not declared on the behavior | `tool.unknown_tool` |
| Tool output violates declared output schema | `tool.invalid_output` |
| Tool body raises | `tool.execution_error` unless it raises `ToolError` |
| Tool loop exceeds `max_tool_turns` | `tool.max_turns_exhausted` |
| Tool-call counter exhausted | `budget.tool_calls_exhausted` |

OpenAI's `finish_reason="tool_calls"` is provider metadata, not a behavior
failure reason. It may appear in `LLMResponse.finish_reason`; it must not become
`behavior.failed.reason`.

## Why the old behavior was wrong

The current `OpenAIProvider.complete` rejects non-empty `tools=` before the SDK
call and reports `llm.network_error`. That was a v1.0.1 compatibility placeholder,
not a real network failure. T4 replaces the placeholder with support; retaining
that reason after implementation would misclassify a supported local shape issue
as external provider/network instability.
