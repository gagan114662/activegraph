## Question

Should `Tool.to_definition` accept `provider="anthropic"|"openai"`, or should
`OpenAIProvider` convert the existing Anthropic/internal shape at request time?

## Pick

**OpenAIProvider converts the existing internal tool definition at request time.**

`Tool.to_definition()` remains the provider-neutral/internal shape already used
by Anthropic:

```python
{"name": name, "description": description, "input_schema": schema}
```

`OpenAIProvider.complete(..., tools=...)` translates each dict into OpenAI chat
completions function shape immediately before the SDK call:

```python
{
    "type": "function",
    "function": {
        "name": tool["name"],
        "description": tool.get("description", ""),
        "parameters": tool.get("input_schema", {"type": "object", "properties": {}}),
    },
}
```

No runtime call site changes. No new public parameter on `Tool.to_definition`.

Commit subject: `Frame t4 amendment D-1: provider-adapter-boundary — keep Tool definition stable; OpenAIProvider translates to function shape at request time`

## Why this boundary

The provider owns wire format. Anthropic consumes the current shape directly;
OpenAI needs `{type:"function", function:{...}}`. Making `Tool` accept a
provider argument would push provider vocabulary into the public tool API and
force every future provider to become another branch inside `Tool`.

Keeping `Tool.to_definition()` stable also preserves every existing Anthropic
fixture and every runtime caller that simply gathers `[t.to_definition() for t in
behavior.tools]`. The only new code belongs in `activegraph/llm/openai.py`, near
the existing `_message_to_openai` and `_extract_text` adapter helpers.

## Binding implementation notes

- Add a private OpenAI helper, e.g. `_tool_to_openai(tool: dict[str, Any])`.
- `OpenAIProvider.complete` must stop rejecting non-empty `tools=`.
- When `tools` is non-empty, `kwargs["tools"]` is the translated OpenAI list.
- Anthropic remains byte-compatible: `AnthropicProvider.complete` still forwards
  the unmodified internal definitions.
- Runtime stays provider-agnostic: it still passes the same `tools` list to any
  `LLMProvider`.
- `_message_to_openai` must also support assistant messages carrying
  `LLMMessage.tool_calls` by emitting OpenAI assistant `tool_calls` entries with
  `function.arguments` as canonical JSON.

## Rejected option

`Tool.to_definition(provider=...)` was rejected because it makes provider shape a
public `Tool` concern, widens the typing surface, and invites drift when the next
provider needs a third schema. A provider adapter is smaller and matches the
existing OpenAI message adapter pattern.
