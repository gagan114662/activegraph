## Question

Specify the raw OpenAI response fields the adapter reads, the `ToolCall`
id/name/args mapping, and how invalid JSON or missing arguments map into
existing reason semantics.

## Pick

Read `choices[0].message.tool_calls` from the OpenAI chat completions response.
Each entry maps to the existing `activegraph.llm.types.ToolCall`:

| OpenAI raw field | ActiveGraph field |
|---|---|
| `tool_call.id` | `ToolCall.id` |
| `tool_call.function.name` | `ToolCall.name` |
| `tool_call.function.arguments` | JSON-decoded `ToolCall.args` |

Only entries with `tool_call.type == "function"` participate. Unknown future
tool-call types are ignored rather than inventing new runtime semantics.

## Argument parsing

OpenAI returns `function.arguments` as a JSON string.

- Valid JSON object -> `ToolCall.args = dict(parsed)`.
- Missing, `None`, or empty string -> `ToolCall.args = {}`.
- Valid JSON non-object -> `ToolCall.args = {"_raw": parsed}`.
- Invalid JSON string -> `ToolCall.args = {"_raw": original_string}`.

The provider does not validate a tool's schema. Runtime already owns input
validation in `_invoke_tool`; malformed or incomplete args must flow to the
existing `tool.invalid_input` path when the declared tool schema rejects them.

This mirrors the existing Anthropic adapter's narrow tolerance for string args:
it tries JSON, then preserves raw content instead of raising a provider-only
exception.

## Finish and parsing behavior

- If extracted tool calls are non-empty, `LLMResponse.tool_calls` is populated
  and `parsed` remains `None`; runtime dispatches tools and recalls the model.
- `finish_reason` is copied from `choices[0].finish_reason` verbatim, commonly
  `"tool_calls"` for the tool-turn response.
- Final non-tool responses keep the existing structured-output path:
  `_extract_text(raw)` then `_parse_structured(...)` when `output_schema` is set.

## Missing identity fields

- Missing `id` maps to `ToolCall.id = ""`; the following tool-result message
  will carry an empty `tool_call_id`, preserving evidence instead of creating a
  new reason.
- Missing `function.name` maps to `ToolCall.name = ""`; runtime treats it as an
  undeclared tool and emits existing `tool.unknown_tool`.
