## Question (verbatim from yaml line 54–63)

Allowlist semantics — flags promised in docs but pending implementation
(e.g. `--set` pending T3) need a way to be excluded without silencing
drift detection wholesale. Options: (a) inline `<!-- drift-allow: --set -->`
HTML comments adjacent to each doc occurrence, (b) a single
`cli_flag_drift_allowlist.toml` file with rationale + expiry note per
entry, (c) refuse to allowlist and require T3 to ship first.

## Pick

**(b) — `cli_flag_drift_allowlist.toml` with rationale + expiry per entry.**

Commit subject: `Frame t2 amendment D-2: toml — central allowlist with mandatory rationale + expiry forces staleness pressure and one-place auditor visibility`

`activegraph/cli_flag_drift_allowlist.toml:rw` permission at yaml line
73 **stays** (the conditional resolves to "include").

## Justification

### Staleness failure mode (verbatim answer to the yaml's ask)

Each entry carries an `expiry_commit_ref` (frame id or commit hash that,
once landed, MUST trigger removal) and an `expiry_date` (hard date
backstop). The gate enforces both:

1. **Hard expiry:** if `expiry_date < today`, the gate fails closed with
   `AllowlistEntryExpiredError(flag=<name>, expired_on=<date>)`. The
   entry cannot be silently extended — extension requires a commit that
   bumps the date, which surfaces in the next PR review.

2. **Resolution expiry:** if `expiry_commit_ref` is the id of a closed
   frame (e.g., `t3-implement-cli-set-flag`), the gate fails closed with
   `AllowlistEntryResolvedError(flag=<name>, resolving_frame=<id>)`.
   This catches the "T3 shipped, we forgot to remove `--set` from the
   allowlist" case.

3. **Forgery resistance:** entries missing either `rationale` or
   `expiry_date` cause the gate to fail closed at load time
   (`AllowlistEntryMalformedError(flag=<name>)`). The schema cannot be
   silently softened by a careless contributor.

Staleness pressure is therefore mechanical, not social. The gate
re-fails every PR after expiry, forcing either resolution or explicit
extension — no "we'll get to it" decay path.

### Visibility guarantee for pending claims (verbatim answer to the yaml's ask)

Each pending claim is at one path: `activegraph/cli_flag_drift_allowlist.toml`.
This guarantees:

- **Diff-reviewable:** every PR that adds, modifies, or removes a
  suppression touches this file. Code review sees the full set.
- **Greppable:** `grep "^\[\[" cli_flag_drift_allowlist.toml | wc -l`
  gives the suppression count. CI can publish this number.
- **Auditor-readable:** a single file beats 8+ markdown files scattered
  across `docs/`. An auditor asking "what drift are we currently
  tolerating?" reads one file, not the whole doc tree.
- **Tooling-amenable:** a `scripts/list_drift_allowlist.py --expiring-soon`
  helper is one TOML parse away. Inline HTML comments would require
  cross-document scanning to answer the same question.

## Schema (informational, not binding on Code Owner)

```toml
# activegraph/cli_flag_drift_allowlist.toml
schema_version = "1"

[[entry]]
flag = "--set"
rationale = "Promised in 8 markdown files per GAUNTLET T2 line 52; implementation pending in frame t3-implement-cli-set-flag."
expiry_commit_ref = "frame:t3-implement-cli-set-flag"
expiry_date = "2026-07-01"
appears_in = [
  "docs/explanation/fork-and-replay.md",
  "docs/how-to/fork-with-overrides.md",
  # ... full list of 8 paths
]

[[entry]]
flag = "--memo"
rationale = "Promised alongside --set in same docs; impl tied to t3."
expiry_commit_ref = "frame:t3-implement-cli-set-flag"
expiry_date = "2026-07-01"
appears_in = [ ... ]

[[entry]]
flag = "--search"
rationale = "Same cluster as --set / --memo; impl tied to t3."
expiry_commit_ref = "frame:t3-implement-cli-set-flag"
expiry_date = "2026-07-01"
appears_in = [ ... ]
```

## Rejected: (a) inline HTML comments

Three failure modes that disqualify this option:

1. **Decay-by-relocation:** when a doc is restructured, the comment
   travels with the doc text — but if the doc is split across files
   or the example moves, the suppression silently follows unrelated
   content. Inline suppressions accumulate without review pressure.

2. **No expiry mechanism:** HTML comments are static strings. Adding
   an `expires=` attribute is possible but requires custom parsing on
   top of standard markdown — at which point we've reinvented (b)
   without the central-file benefit.

3. **Auditor blindness:** "what drift are we tolerating right now?"
   requires scanning every `.md` file in the corpus. Cannot be
   answered from a single file diff.

The locality benefit (suppression travels with the doc) is real but
small — every PR that touches a suppression already touches the file
that contains it, so review proximity is already guaranteed at the
PR-diff level regardless of whether the suppression lives in the doc
or in a TOML.

## Rejected: (c) refuse to allowlist, ship T3 first

Inverts the gauntlet's intended sequence. Per the frame's goal at line
22–25, t2 must "pass against current main with an explicit allowlist
for T3-pending flags" — main today has `--set` / `--memo` / `--search`
in docs but not in code. Refusing the allowlist forces one of:

- **t2 cannot ship until t3 ships:** sequencing inversion. t2's purpose
  is to *catch* this drift class going forward; deferring it until t3
  resolves the current instance leaves no gate in place when the next
  instance happens.
- **Delete the doc references first:** scrubs evidence of the
  intended user-facing surface from the docs, harming roadmap signal
  for users reading current docs.

Neither is acceptable. The allowlist mechanism is necessary; the only
question is which form, and (b) wins on staleness pressure and
auditor visibility.

## Implementation hints (footnote per Frame Architect review)

`AllowlistEntryResolvedError` requires the gate to know when a referenced
frame has closed. Two ways to wire this coupling, both acceptable —
**Code Owner picks during impl, but the choice MUST be documented in
the gate's docstring so the dependency is not hidden**:

**(i) Read frame status from `frames/<id>.status`** (or equivalent
frame-lifecycle metadata file). Tight coupling between the drift gate
and the frame-lifecycle subsystem. Authoritative — the gate knows
exactly when a frame has closed.

- Pros: zero false negatives; gate fires the moment the frame closes.
- Cons: drift gate depends on the existence and schema of
  `frames/<id>.status`. If the frame-lifecycle metadata schema changes,
  the gate breaks.
- Mitigation: pin the status-file shape in CONTRACT v1.1 when the
  allowlist schema is locked there. Drift gate then has a contract
  to read against.

**(ii) Trust a simpler signal — frame yaml present + `status: closed`
field grep.** Loose coupling. Gate only knows what's in the yaml itself.

- Pros: no dependency on a separate `.status` file; one fewer artifact
  per frame to maintain; degrades gracefully if frame-lifecycle subsystem
  is refactored.
- Cons: may lag a frame's actual closure by one commit (if the yaml's
  status field is updated in a separate commit from the closing work).

**Default recommendation:** pick (i) iff the team commits to writing
`frames/<id>.status` as part of frame-closure ritual (current practice
per `t1a` closure: `outer:6b02680` includes status flip). Otherwise
pick (ii). Either way, the docstring on the gate script MUST cite the
mechanism so an auditor reading the gate code knows where the
"closed?" determination comes from — silent magic is the failure mode
to avoid.

This is an implementation detail Code Owner decides, not a Spec
amendment. Flagged here so it doesn't become a hidden assumption
between Spec and Code.
