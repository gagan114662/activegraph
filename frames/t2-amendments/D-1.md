**Decision class:** implementation — affects gate accuracy and future maintenance
**Status:** PROPOSED — awaiting Frame Architect review before D-2 begins

## Question (verbatim from yaml line 44–51)

Source-of-truth for CLI flags — does the gate extract flags by importing
the CLI module and walking the argparse/click parser (dynamic), or by
static grep for `--<name>` patterns across `activegraph/cli/*.py` (static)?

## Pick

**Static.** Pattern-match call-site form `add_argument("--<name>"` and
`@(?:click\.)?option("--<name>"` (plus typer's `--<name>` in `typer.Option`)
across `activegraph/cli/**/*.py`. No import, no parser walk.

Commit subject: `Frame t2 amendment D-1: static — fail-closed import safety + conditional-flag honesty + framework-stable maintenance shape`

## Justification against the three axes

### (a) Accuracy when a flag is registered conditionally

| Pattern | Static | Dynamic |
|---|---|---|
| `if FEATURE: parser.add_argument("--foo")` | reports `--foo` always | reports `--foo` only when `FEATURE` truthy at gate-run time |
| `for n in subcmds: p.add_argument(f"--{n}-flag")` | misses (f-string, no literal) | reports the resolved flag set |
| Subcommand registered in a plugin imported lazily | reports if the file is under `cli/` | misses unless plugin is imported by gate-run time |

Static **over-reports** the conditional case; dynamic **under-reports** it.
Over-reporting is the correct failure direction for a drift gate: if a
flag exists in code under any condition, docs that reference it are not
drifted, and docs that don't reference it but should are correctly
flagged. Under-reporting silently lies about coverage.

The f-string edge case is a known antipattern — no current CLI flag is
constructed dynamically (verified via static read of `cli/quickstart.py`
and `cli/main.py`), and a constraint can pin this forward.

### (b) Failure mode when the CLI module fails to import

| Failure | Static | Dynamic |
|---|---|---|
| SyntaxError in `cli/foo.py` | regex still runs; ast-parse mode would fail with parse error → gate exit nonzero (fail-closed per constraint line 30) | `ImportError` → caught? if caught, gate sees zero flags → "no drift" silently (open) |
| Missing dep in CLI import-time path | regex unaffected | `ImportError` → same silent-zero hazard |
| Decorator side-effect raises | regex unaffected | full module init fails |

The frame's constraint at line 30 is `"gates fail closed (default-deny on
parse errors)"`. Dynamic's natural failure is silent-empty unless the
gate wraps every import in `try/except` and treats `ImportError` as
"drift detection unavailable, fail closed" — which adds code and still
couples gate health to runtime import health. Static decouples them.

### (c) Maintenance cost as new CLI subcommands are added

| Cost | Static | Dynamic |
|---|---|---|
| New file under `cli/` | auto-discovered via glob | needs `__init__.py` import wiring or per-file scan code |
| Switch argparse → click | regex extended by one alternation | full re-implementation of parser walk |
| Add typer-style sub-Typer apps | regex extended by one alternation | recursive parser-tree traversal (typer wraps click which wraps argparse) |
| New CLI maintainer onboarding | reads one regex | learns framework-specific introspection APIs |

Static maintenance is monotonically linear in framework count (one
alternation per framework). Dynamic maintenance is multiplicative
(framework × subcommand-nesting × import-graph).

## Rejected: dynamic

Dynamic's only real advantage is catching dynamically-named flags
(antipattern, none in current codebase). Its disadvantages — silent
failure on import error, fragility to import-graph changes, framework
fragmentation cost — all violate the frame's discipline (fail-closed)
or compound future maintenance.

## Implementation sketch (informational, not binding on Code Owner)

```python
# scripts/gate_cli_flag_drift.py
import re, ast, pathlib

CLI_DIR = pathlib.Path("activegraph/cli")
FLAG_PATTERNS = [
    re.compile(r'add_argument\(\s*["\']--([a-z][a-z0-9-]*)'),
    re.compile(r'@(?:click\.)?option\(\s*["\']--([a-z][a-z0-9-]*)'),
    re.compile(r'typer\.Option\([^)]*["\']--([a-z][a-z0-9-]*)'),
]

def extract_flags() -> set[str]:
    flags: set[str] = set()
    for py in CLI_DIR.rglob("*.py"):
        text = py.read_text()
        # Validate syntax — fail-closed on parse error per constraint
        ast.parse(text, filename=str(py))
        for pat in FLAG_PATTERNS:
            flags.update(pat.findall(text))
    return flags
```

The `ast.parse` call enforces the fail-closed constraint at the syntax
boundary: if any CLI file fails to parse, the gate exits nonzero before
reporting drift — same loud-failure shape as `mypy --strict` on broken
syntax.

## Future framework adoption (footnote per Frame Architect review)

The "one regex alternation per framework" claim holds for **explicit**
`--<name>` literals in source — argparse `add_argument("--foo")`, click
`@option("--foo")`, typer `Option("--foo")`. The current corpus is
argparse + click only (verified by static read of `cli/quickstart.py`
and `cli/main.py`), so the patterns above are sufficient for what ships.

**Typer caveat — flagged for future adopters, not blocking T2:** typer
auto-derives flag names from function parameter names when no explicit
`--name` literal is given. For example:

```python
@app.command()
def cmd(set_flag: str): ...   # typer registers `--set-flag` at runtime,
                              # but no literal "--set-flag" appears in source.
```

Static regex extraction will miss this surface. If typer is ever
adopted in `activegraph/cli/`, the static extractor must add a
typer-specific AST pass that walks `@app.command`/`@app.callback`
decorators and reads the parameter signatures (snake_case → kebab-case
conversion is typer's default). Code Owner should treat this as a
known future extension, not a bug in the initial static pass.

This is a maintenance-cost note against axis (c), not a re-litigation
of the pick — dynamic introspection has the *same* coverage problem
for any framework that registers flags programmatically without
literal strings, and the fail-closed shape of static is still the
correct draw of the line for the corpus that ships.

## Cross-amendment ripple (signals only, not pre-commitments for D-2)

D-2 is orthogonal to extraction approach. The allowlist semantics
question doesn't change based on whether flags come from regex or
parser-walk — once the set of CLI-known flags is determined, allowlist
applies the same way. Static does, however, make the synthetic-drift
fixture easier to write (a fixture file with a `parser.add_argument`
call is sufficient; no need to stand up an importable CLI plugin).
